# Ограничения MVP и дальнейшее развитие

Shine 0.2.0 включает многомодульные программы, `import`, явный `export`, source manager, HIR-linking, простые классы, расширенную встроенную математику и упорядоченный тип `Dictionary`. Остальные возможности по-прежнему намеренно ограничены.

## Что входит в MVP

- числа, строки, логические значения и `none`;
- динамические и фиксированно типизированные переменные;
- `const`;
- списки, индексы, срезы и методы;
- динамические и типизированные словари;
- деструктуризация;
- выражения и сравнения;
- функции и `return`;
- `if / else if / else`;
- четыре формы `loop`;
- диапазоны и `step`;
- `break` и `continue`;
- console I/O;
- текстовые файлы;
- математика и базовые агрегаты;
- диагностика;
- CLI для создания, запуска, проверки, сборки, форматирования и тестов.

## Чего пока нет

### Система языка

- package manager и registry;
- overload;
- generics;
- macros;
- inheritance, interfaces и abstract classes;
- исключений и `try/catch`;
- pattern matching;
- отдельного `while` или `for` — используется только `loop`;
- ternary expression;
- полноценных chained comparisons;
- короткой формы функции.

### Коллекции и научные типы

- отдельного Tuple;
- отдельного Set;
- Vector;
- Matrix;
- Complex;
- Fraction;

Для простых tuple используется `const List` и деструктуризация. Для уникальных значений — `unique()`.

### Платформы и runtime

- Web и browser API;
- GUI;
- AI API;
- WebAssembly;
- threads;
- async runtime;
- сетевого API;
- debugger;
- полноценного LSP;
- self-hosting;
- LLVM/native backend.

## Текущие практические границы

- комментарий `//` должен начинать отдельную логическую строку;
- `build` создаёт evaluator bundle из двух соседних файлов;
- `run`, `check`, `build` принимают entry-файл и автоматически загружают его модули;
- formatter минимален и ориентирован на стандартный стиль;
- файлы поддерживают только чтение и полную запись текста;
- срезы только прямые, без шага;
- шаг диапазона только целый;
- типы динамических выражений окончательно проверяются runtime;
- функции рекомендуется объявлять только на верхнем уровне.
- быстрый числовой VM покрывает скалярные вычисления, условия, циклы, вызовы функций и однородные списки `Int`/`Float`; строки, объекты и смешанные списки выполняются общим evaluator до появления native backend.

## План развития

Предполагаемый порядок:

1. завершение HIR и семантического resolver;
2. ООП, visibility и модель ошибок;
3. typed MIR/SSA и LLVM native AOT;
4. async runtime и server stack;
5. Array CPU/CUDA/Metal;
6. DataFrame/Arrow и scientific libraries;
7. ML, notebook и package ecosystem;
8. LSP, debugger, profiler и стабилизация 1.0.

Будущие возможности должны сохранять простой синтаксис и понятные ошибки.

## Как узнать, поддерживается ли возможность

1. Проверьте [полный справочник](18-language-reference.md).
2. Запустите `shine check file.shn`.
3. Сверьтесь с master specification в корне репозитория.
4. Не считайте roadmap уже реализованной функцией.
