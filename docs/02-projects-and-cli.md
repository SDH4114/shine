# Проекты и команды CLI

Исполняемая команда языка называется `shine`.

## Список команд

| Команда | Назначение |
|---|---|
| `shine new <project>` | Создать новый проект |
| `shine run <file.shn>` | Выполнить программу |
| `shine check <file.shn>` | Проверить программу без выполнения |
| `shine build <file.shn>` | Создать исполняемый bundle |
| `shine fmt <file.shn>` | Отформатировать файл на месте |
| `shine test [project]` | Выполнить файлы `tests/*.shn` |
| `shine help` | Показать справку |
| `shine version` | Показать версию |

Короткие варианты справки и версии:

```bash
shine -h
shine --help
shine -V
shine --version
```

## `shine new`

```bash
shine new calculator
shine new ./tools/calculator
shine new /absolute/path/calculator
```

Относительный путь считается от текущей папки. Недостающие родительские каталоги создаются автоматически, а целевой каталог проекта должен отсутствовать. Shine не перезаписывает существующий проект.

Файл `shine.toml`:

```toml
[project]
name = "calculator"
version = "0.1.0"
entry = "src/main.shn"
```

В версии 0.1.3 команды принимают entry-файл и автоматически разрешают его imports. Чтение `entry` напрямую из `shine.toml` будет добавлено вместе с полноценным project loader.

## `shine run`

```bash
shine run src/main.shn
```

Команда строит граф модулей, понижает его через HIR-linker и запускает связанную программу через reference tree-walking backend. Обычный вывод идёт в stdout. Ошибки идут в stderr и завершают процесс ненулевым кодом.

Пути в `readFile` и `writeFile` считаются от текущей рабочей папки процесса, а не от папки исходного файла.

## `shine check`

```bash
shine check src/main.shn
```

Статическая проверка анализирует все тела функций и ветки, но не вызывает `input`, не читает и не записывает файлы и не выполняет пользовательские вычисления. Она проверяет:

- синтаксис;
- существование имён;
- совместимость известных типов;
- аннотированные параметры и результаты функций;
- количество аргументов пользовательских функций;
- изменение `const`;
- `break` и `continue` вне цикла;
- выражения внутри строковой интерполяции.

Динамические значения могут быть неизвестны во время проверки. Операции с ними окончательно проверяются при запуске.

## `shine build`

```bash
shine build src/main.shn
```

Результат для однофайлового `main.shn`:

```text
target/shine/
├── main
└── main.shn
```

Запуск:

```bash
./target/shine/main
```

Оба файла должны оставаться рядом. Это исполняемый bundle tree-walking backend, а не оптимизированный native-бинарник. Подробнее: [Форматирование и сборка](17-formatting-and-build.md).

Для многомодульной программы рядом с executable создаётся `main.shine-src/` с entry marker и всеми использованными модулями. LLVM native AOT заменит этот временный формат на standalone native artifact.

## `shine fmt`

```bash
shine fmt src/main.shn
```

Перед изменением файла formatter проверяет синтаксис. Он приводит отступы к четырём пробелам и удаляет завершающие `;`, поскольку они необязательны.

## `shine test`

Из корня проекта:

```bash
shine test
```

Из другой папки:

```bash
shine test /path/to/project
```

Команда выполняет все `.shn`-файлы непосредственно внутри `tests/` в алфавитном порядке. Первый сбой завершает тестовый запуск с ошибкой.

## Коды завершения

- `0` — команда выполнена успешно;
- `1` — ошибка CLI, файла, синтаксиса, типов или выполнения.

Это позволяет использовать Shine в shell-скриптах и CI:

```bash
shine check src/main.shn && shine test
```
